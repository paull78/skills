# A Pattern Language — Christopher Alexander

## Core Thesis

Good design is not invented from scratch — it emerges from a language of interconnected patterns, each solving a recurring problem in context. Patterns are not templates; they are relationships between context, problem, and solution that produce life in the whole.

## Principles Applied to Software

- **Patterns form a language.** Individual patterns are words; the connections between them form grammar. A design that uses patterns in isolation — without understanding how they connect — is incoherent.
- **Each pattern resolves a force.** A pattern exists because competing forces create tension. The pattern is the resolution. If you can't name the forces, you don't have a pattern — you have a habit.
- **Context is everything.** A pattern that is right in one context is wrong in another. Applying patterns without understanding the context they serve produces dead structure.
- **Wholeness over parts.** The quality of a system comes from the relationships between its parts, not the quality of individual parts. A system of mediocre parts that relate well outperforms brilliant parts that don't.
- **Emergent order, not imposed order.** Good structure unfolds step by step, each decision informed by what came before. Top-down master plans produce rigid, fragile systems.
- **Centers strengthen centers.** Each component should reinforce the components around it. If adding a component weakens its neighbors, the design is wrong.

## Pattern Structure

Every pattern follows: **Context → Forces → Solution → Resulting context.**

When evaluating a design, ask for each decision:
1. What is the context?
2. What forces are in tension?
3. How does this resolve them?
4. What new context does it create for the next decision?

## Key Anti-Patterns

- Applying a pattern because it has a name, not because the forces demand it.
- Patterns in isolation — no connections between them.
- Imposed symmetry or uniformity that doesn't serve the context.
- Over-specifying structure instead of allowing it to emerge from use.
- Designing for the designer's aesthetics rather than the inhabitant's (user's) experience.

## Grilling Questions

- What forces are in tension here, and how does this design resolve them?
- Do these patterns connect into a coherent language, or are they isolated choices?
- Does this structure serve the people who will live in it (use it), or the people who built it?
- Is this order emerging from the problem, or being imposed on it?
- Does adding this component strengthen or weaken its neighbors?
